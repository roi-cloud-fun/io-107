# IO-107 SDLC Pipeline — Instructor Pre-Class Setup

> **Generated by labforge untagged on 2026-05-25.** Do not hand-edit — re-run `generate_setup_artifacts.py` if the lab content changes.

This checklist gets a fresh AWS account ready to deliver **IO-107 SDLC Pipeline** to a cohort of students. Work through it once per region you plan to deliver in. Budget **~45 minutes of hands-on time** plus **~30 minutes of AWS provisioning wait** the first time; subsequent cohorts are quicker because the shared training environment persists.

The work splits cleanly into two phases:

1. **Shared infrastructure** (one-time per region) — you, the instructor, run `lab_environment/training_env/`.
2. **Per-student bootstrap** (once per student per cohort) — either you run it on behalf of students, or each student runs it themselves at the start of class.

---

## Phase 0 — Backend bootstrap (one-time per account)

Terraform state must live somewhere durable + locked. **Terraform 1.10+ supports native S3 locking** (`use_lockfile = true`) — no DynamoDB lock table needed. The only thing to provision is an S3 bucket:

```bash
# Replace REGION as needed
aws s3 mb s3://io107-tfstate-REGION --region REGION

aws s3api put-bucket-versioning \
  --bucket io107-tfstate-REGION \
  --versioning-configuration Status=Enabled
```

> `aws s3 mb` is the higher-level make-bucket helper — it handles the `LocationConstraint` quirk for non-us-east-1 regions internally, so you don't have to. The `put-bucket-versioning` step still uses `s3api` because the higher-level `aws s3` subcommand doesn't expose versioning configuration.

Then add a backend block to `training_env/main.tf` (and `student_bootstrap/main.tf` if you're in split mode). **The generator intentionally does not write the backend block** — it varies per account.

```hcl
terraform {
  backend "s3" {
    bucket       = "io107-tfstate-REGION"
    key          = "training_env/terraform.tfstate"
    region       = "REGION"
    encrypt      = true
    use_lockfile = true   # Terraform 1.10+ native locking
  }
}
```

> The legacy `dynamodb_table` lock backend was deprecated in Terraform 1.10 (Sep 2024) — S3 conditional writes now provide the same lock guarantees without the extra DDB table to provision + pay for.

## Phase 1 — Shared training infrastructure

Provisioned once per region. Multiple cohorts can reuse it.

### What gets created

This list is auto-derived from the IO-107 SDLC Pipeline labs' LabForge setup-gap analysis (`INSTRUCTOR_SHARED_PER_REGION` resources):

- `aws_eks_cluster` **training-eks-cluster** — Single EKS cluster per region — namespaces isolate students
- `aws_ecr_repository` **<ECR repository>** — Shared image registry — per-student tags
- `aws_iam_role` **io107-** — Shared regional resource
- `aws_db_subnet_group` **training** — VPC-level — shared across all RDS in the region
- `aws_kms_key` **training_rds** — Account-level encryption keys
- `aws_security_group` **training_db** — VPC-level — shared across all workloads

### Steps

1. `cd lab_environment/training_env/`

2. Copy and edit the tfvars:

    ```bash
    cp terraform.tfvars.example terraform.tfvars
    # confirm aws_region, vpc_cidr, eks_cluster_name match your environment
    ```

3. Configure the backend:

    ```bash
    cat > backend.tf <<'EOF'
    terraform {
      backend "s3" {
        bucket       = "io107-tfstate-REGION"
        key          = "training_env/terraform.tfstate"
        region       = "REGION"
        encrypt      = true
        use_lockfile = true
      }
    }
    EOF
    ```

4. Initialise and apply:

    ```bash
    terraform init
    terraform plan -out=tfplan
    terraform apply tfplan
    ```

    EKS cluster provisioning takes ~15 minutes. Plan accordingly.

5. **Complete the CodeStar source connection handshake** (one-time, per region):

    - Open the AWS Console → CodePipeline → Settings → Connections
    - Click the `io107-source` connection
    - Click **Update pending connection**
    - Authorise the OAuth flow with the course's source provider (GitHub)

    Until this step is complete, every student pipeline's Source action will fail with "Connection is pending."

6. **Confirm** outputs printed by Terraform — `eks_cluster_name`, `codestar_connection_arn`, `codebuild_service_role_arn`, etc. Save these somewhere visible (lab whiteboard, cohort Slack pinned message). Students reference them indirectly via `terraform_remote_state`, but having them visible helps when troubleshooting.

## Phase 2 — Per-student bootstrap

Provisioned once per student per cohort. Three deployment options — pick one based on your delivery model:

### Option A: Instructor pre-provisions (recommended for ≥10 students)

Loop over the cohort roster before class:

```bash
cd lab_environment/student_bootstrap/
terraform init -backend-config="key=student_bootstrap/$STUDENT.tfstate" -reconfigure

for STUDENT in alice bob carol; do
  terraform workspace new $STUDENT 2>/dev/null || terraform workspace select $STUDENT
  terraform apply -auto-approve \
    -var="student_id=$STUDENT" \
    -var='training_env_remote_state={bucket="io107-tfstate-REGION",key="training_env/terraform.tfstate",region="REGION"}'
done
```

Students arrive to class with their pipeline already wired. Each lab's "Pre-Lab Setup" section in the student-facing markdown tells students to skip the bootstrap and proceed directly to Task 1.

### Option B: Students self-provision (recommended for ≤10 students with strong AWS background)

Hand each student:

- The course's Git repo containing `lab_environment/student_bootstrap/`
- Their AWS credentials (scoped to a per-student IAM role)
- The values for `training_env_remote_state` (bucket + key + region)
- A unique `student_id`

Each student runs the steps in their lab's **Pre-Lab Setup** section before Task 1.

### Option C: Hybrid (instructor pre-provisions the "spine," student adds per-lab pieces)

Useful for variable-length cohorts. Instructor pre-provisions Phase 1 + the K8s namespaces + S3 buckets + IRSA roles. Students provision their own pipelines on day-of, which takes ~2 minutes per student.

## Pre-class verification

The day before class, verify the environment by spinning up a "smoke test" student and walking Lab 1 end-to-end yourself:

```bash
terraform apply -var="student_id=smoke-test" ...
git clone https://github.com/jessetop/io107-lab1-eks-app
# ... follow Lab 1 from the student MD
```

If Lab 1's `helm upgrade --install` lands a healthy Deployment in the `lab1-smoke-test` namespace and the LoadBalancer ELB returns 200, your environment is ready.

Then tear down the smoke-test student:

```bash
terraform destroy -var="student_id=smoke-test" ...
```

## Cleanup between cohorts

- `terraform destroy` against `student_bootstrap/` for each student workspace clears pipelines + buckets + namespaces + (if Lab 4) Aurora clusters.
- The shared `training_env/` apply stays in place across cohorts — EKS cluster, ECR, KMS keys persist. Only re-apply if course content changes.

## Troubleshooting

### "Pipeline stuck in 'Source' stage with 'Connection is pending'"

Phase 1 step 5 was skipped — complete the CodeStar OAuth handshake in the AWS Console.

### "terraform apply fails on student_bootstrap with 'no matching credentials'"

The student's role doesn't have permissions on the training_env state bucket. Add `s3:GetObject` on `arn:aws:s3:::io107-tfstate-REGION/training_env/terraform.tfstate` to the per-student IAM policy.

### "Aurora Blue/Green fails with 'cluster identifier already in use'"

Two students chose the same `student_id`. Identifiers must be unique within the AWS account.

### "kubectl reports 'You must be logged in to the server (Unauthorized)'"

The student's role isn't in the EKS cluster's `aws-auth` ConfigMap. Add their per-student role to the ConfigMap via `kubectl edit configmap aws-auth -n kube-system`. (LabForge does not manage this — it's an EKS day-1 step.)
